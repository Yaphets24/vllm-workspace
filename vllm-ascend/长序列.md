## 1. 总体关系

长序列推理里，最核心的不是某一个单独特性，而是一组互相依赖的机制：

```text
KV Cache 管理
├─ Prefix Caching
└─ Chunked Prefill
```

它们之间的关系可以概括为：

- `KV cache` 是底座，负责 block 的分配、回收、引用计数、映射和物理存储组织
- `prefix caching` 建立在 KV cache 之上，负责“复用已经算过的前缀”
- `chunked prefill` 也建立在 KV cache 之上，负责“把仍然需要计算的长 prompt 分多轮推进”

一句话记忆：

- `KV cache` 负责“存什么、怎么管”
- `prefix caching` 负责“少算”
- `chunked prefill` 负责“分开算”

## 3. 三者的统一视图

### 3.1 从“一个请求”看

一个长请求进入系统后，大体会经历：

```text
请求进入
-> Request 创建
-> 生成 block_hashes（若需要）
-> 查询本地/外部 KV 是否可复用
-> 分配或复用 blocks
-> 写入 block_table
-> 计算 slot_mapping
-> worker forward
-> 更新 KV cache 状态
-> 继续执行或释放
```

在这个总流程里：

- 是否“查询可复用前缀”，对应 prefix caching
- 是否“本轮只推进一部分 prompt”，对应 chunked prefill

### 3.2 从“一个 block”看

一个 block 的生命周期大体是：

```text
空闲块
-> 被请求分配
-> 被写入 KV
-> 可能成为 cached block
-> 请求结束后 ref_cnt 降为 0
-> 进入 free queue
-> 后续要么被 prefix hit 复用，要么被 eviction 后重分配
```

这里：

- prefix caching 依赖的是 “cached block + block hash”
- chunked prefill 依赖的是 “请求在多轮中不断追加 block”

### 3.3 从“长 prompt”看

对于一个长 prompt，有三种典型情况：

#### 情况 A：无 prefix hit，无 chunked prefill

```text
整段 prompt 一次完整 prefill
```

#### 情况 B：有 prefix hit，无 chunked prefill

```text
前缀跳过
剩余 suffix 一次完整 prefill
```

#### 情况 C：有 prefix hit，且有 chunked prefill

```text
前缀跳过
剩余 suffix 仍然很长
-> 再分成多个 chunk 多轮 prefill
```

这也是长序列场景里最常见、最有价值的组合方式。

## 4. `DeepSeek-V3.1` 这条主线该怎么理解

在这个工作区里，`DeepSeek-V3.1` 是非常好的学习样本，因为它同时涉及：

- 长上下文
- DeepSeek MLA
- chunked prefill
- prefix caching
- Ascend worker 上的特殊 KV 布局

你可以用下面这句话把它记住：

**对 `DeepSeek-V3.1` 来说，管理层仍然统一按 block 管，但物理层是 MLA 特化 KV 布局；prefix caching 决定哪些 token 不再进入 MLA，chunked prefill 决定剩余 token 如何分批进入 MLA。**

再展开一点：

- `kvcache.md` 解释“块是怎么分、怎么持有、怎么释放的”
- `prefix-caching.md` 解释“哪些块可以直接复用”
- `chunk_prefill.md` 解释“还要算的长后缀怎么分多轮推进”

如果把它们合到一起看，`DeepSeek-V3.1` 的长请求大致就是：

```text
长请求进入
-> block 级 KV 管理生效
-> 若命中 prefix cache，则前缀直接复用
-> 剩余后缀若仍很长，则切成多个 chunk
-> 每轮只把一个 chunk 送入 MLA
-> MLA 内部再对历史 context 做二次 chunk
-> 直到整段 prompt KV 完整建立，再进入 decode
```

## 5. mixed batch 为什么重要

长序列学习里，有一个很重要的交叉点是 mixed batch：

```text
[decode 请求] + [chunked prefill 请求]
```

这会同时牵动三份笔记：

- `kvcache.md`
  因为不同请求各自有自己的 block table 和 slot_mapping
- `chunk_prefill.md`
  因为 prefill 请求只推进一段 chunk
- `prefix-caching.md`
  因为某些请求可能已经有前缀命中

推荐你回顾 mixed batch 时，重点盯住这几个量：

- `req_id_to_index`
- `block_table`
- `slot_mapping`
- `num_computed_tokens`
- `num_scheduled_tokens`
- `attn_state`
- `actual_seq_lengths_q`
- `actual_seq_kv`

只要这几个量能读顺，batch 内 decode/prefill 混跑就不会再抽象。

## 8. 一句话总总结

长序列推理在这个项目里，本质上就是：**先用统一的 block 化 KV cache 管理请求上下文，再通过 prefix caching 尽量复用已有 block，通过 chunked prefill 把仍需计算的长后缀分步推进，而 `DeepSeek-V3.1` 的 MLA 则在这个统一管理框架上采用了特化的物理 KV 布局和内部 context chunking。**
